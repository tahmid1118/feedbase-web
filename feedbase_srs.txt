================================================================================
SOFTWARE REQUIREMENTS SPECIFICATION
Feedbase - Multi-Tenant Feedback & Roadmap Platform
Version 1.1
================================================================================

--------------------------------------------------------------------------------
1. INTRODUCTION
--------------------------------------------------------------------------------

1.1 Purpose
    This document specifies the requirements for Feedbase, a multi-tenant SaaS
    platform that lets companies collect product feedback from their users,
    prioritize it through voting, and communicate progress through a public
    roadmap and changelog.

1.2 Scope
    Each customer (tenant) receives a branded public portal where their end
    users submit and upvote feedback, and a private admin dashboard where the
    tenant's team triages and manages that feedback, plans a roadmap, and
    publishes release notes. The system is delivered as a web application
    backed by a stateless REST API and a relational database.

1.3 Definitions
    Tenant        An isolated customer workspace.
    Visitor/Guest An unauthenticated end user on a tenant's public portal.
    Member        An authenticated tenant team member (owner or user).
    Admin         A platform operator (the app owner) with super-access across
                  every workspace; a separate identity from tenant accounts (the
                  same email may be both).
    Portal        The public, tenant-branded surface (board, roadmap, changelog).
    Dashboard     The private, authenticated workspace application.

--------------------------------------------------------------------------------
2. OVERALL DESCRIPTION
--------------------------------------------------------------------------------

2.1 Product Perspective
    - Frontend: Next.js (App Router) web application; communicates only with the
      REST API.
    - Backend: Node.js/Express REST API over a MySQL database (raw SQL, no ORM).
    - One codebase serves two surfaces: the authenticated dashboard and the
      public, per-tenant portal.

2.2 User Roles & Permissions
    There are exactly three roles.
    - Visitor/Guest: view a tenant portal, submit feedback (name optional, but a
      contact email is required for follow-up), and upvote posts. No account
      required.
    - User: an authenticated workspace member with baseline access (Profile only
      in settings). Members join by invitation.
    - Owner: the workspace administrator — manages team, tags, integrations, API
      keys, audit logs, branding, billing, and feedback deletion. A workspace has
      exactly ONE owner (its creator); everyone else is a member. An account may
      therefore own some workspaces and be a member of others; the two are shown
      separately in the workspace switcher.
    - Admin (platform): the app operator. Stored separately from tenant accounts,
      signs in through a dedicated admin sign-in, and has super-access over every
      workspace and user through the Admin Panel (see 3.11). "Admin" is never a
      role inside a workspace.

2.3 Multi-Tenancy
    - Each tenant is reachable at <subdomain>.<root-domain> and, optionally, a
      custom domain.
    - All data is isolated by tenant; every query is tenant-scoped.
    - Reserved subdomains (www, app, admin, ...) are never tenants.

2.4 Operating Environment
    - Modern evergreen web browsers.
    - Cloud-hosted application and API; managed MySQL; wildcard DNS
      (*.<root-domain>).

--------------------------------------------------------------------------------
3. FUNCTIONAL REQUIREMENTS
--------------------------------------------------------------------------------

3.1 Authentication & Accounts
    - Email/password authentication using JWT bearer tokens (OAuth-ready).
    - The session carries the user identity, tenant, and role.
    - Profile management: name, contact, avatar upload, and password change.
    FR-D1  Delete account  A user may permanently delete their account from
                          Settings. Before anything happens they are shown exactly
                          what will be lost, and must re-enter their password and
                          type a confirmation word. Workspaces they OWN are deleted
                          with the account (all feedback, roadmap, changelog and
                          member access; any subscription is cancelled), because a
                          workspace has exactly one owner and cannot be
                          transferred. Workspaces they merely joined are retained:
                          only their membership is removed, and their posts and
                          comments there remain but become anonymous.
    FR-I1  Invitations .. A workspace owner invites teammates by email address.
                          The address is validated (format + mail-domain lookup)
                          before sending. The invitee receives a branded email with
                          a secure link that is valid for 7 days and can be used
                          only once; the owner can revoke a pending invitation,
                          which disables its link immediately. Accepting as a new
                          person opens a sign-up dialog (email fixed), after which
                          they create their own workspace and are taken to the
                          workspace that invited them. An existing account must be
                          signed in as the invited address and joins in one step.
                          Invitees always join as members. Invitations count
                          against the plan's team-member limit (see FR-B5).
    FR-S1  Single-session  On the Free and Pro plans an account may be signed in
           login .......... on only ONE device or browser at a time. Attempting to
                          sign in elsewhere while a session is live is refused, and
                          the user is told to sign out on the other device first
                          (with the option to upgrade). Signing out releases the
                          session immediately. A session that has been inactive for
                          15 minutes is treated as abandoned — the next sign-in
                          takes it over and ends the old one — so closing the
                          browser without signing out never locks a user out. Once
                          a session ends, that device is signed out on its next
                          action rather than being left in a broken state.
                          Switching workspaces does not count as a new device.
                          BUSINESS plans permit unlimited concurrent devices,
                          browsers and tabs (see FR-B5).
    FR-S2  Single tab .... On single-session plans, opening the dashboard in a
                          second tab or window of the same browser shows a blocking
                          notice rather than a second working copy, with an option
                          to upgrade. Closing the original frees the other tab
                          automatically.
    FR-W1  Workspaces ... One account (email) may belong to multiple workspaces
                          (tenants), and only ever sees workspaces it belongs to.
                          A new account starts with NO workspace and is sent to
                          onboarding to create its first (name, subdomain,
                          optional website). Users can create additional
                          workspaces and switch the active one from the sidebar;
                          the dashboard re-scopes to the selection without a
                          re-login. Each new workspace is owned by its creator and
                          seeded with the default, status-linked roadmap columns.
                          Each workspace subdomain must be unique; the create
                          forms validate availability live before submission.

3.2 Public Portal (Visitors)
    The portal is the primary channel for collecting feedback.
    - Branded header (tenant logo, or a brand-colored initial fallback) using
      the tenant's primary color.
    FR-P1  Board ........ Browse feedback posts with vote counts, tags, status,
                          and comment counts. Sortable by newest, oldest, most
                          upvoted, or least upvoted (pinned posts stay on top).
    FR-P2  Submit ....... Any visitor may create a post (title and a contact
                          email required; details, type and name optional). The
                          email is mandatory so the team can reach the submitter
                          with updates on their post; it is never shown publicly.
                          A logged-in submitter is identified by their account
                          instead. On workspaces whose plan includes attachments
                          (see FR-A1), the submit form also lets the visitor
                          attach photos or a short video.
    FR-P3  Vote ......... Any visitor may upvote/remove an upvote. Voting is
                          limited to one vote per browser per post via a
                          persistent cookie identifier; no account required.
    FR-P4  Post detail .. Read a post and its threaded comments.
    FR-P5  Comment ...... Any visitor may add a comment, or a threaded reply to a
                          comment, on a post — anonymously or with an optional
                          name/email (the name is remembered for the session). A
                          logged-in user's comment/post is attributed to them,
                          showing their name and profile picture instead of a
                          guest identity. An anonymous guest who leaves no name is
                          shown with a stable, friendly pseudonym and a coloured
                          incognito avatar (consistent across that guest's
                          posts/comments, distinct from other guests, and clearly
                          marked as anonymous) rather than a uniform "Anonymous".
    FR-P6  Share ........ Each post has a public share control: copy link, the
                          device's native share sheet, and one-click social
                          intents (X, LinkedIn, Facebook, WhatsApp, Email). Shared
                          links unfurl with a rich preview (Open Graph / Twitter
                          Card) including a branded, per-post preview image.
    FR-P7  Own content .. A logged-in user may edit or delete their own posts and
                          comments from the public portal (owner-only; enforced
                          server-side). Guests cannot edit/delete.
    FR-A1  Attachments .. On workspaces whose plan permits it (Pro and above; see
                          FR-B5), a feedback post may carry up to three photo or
                          short-video attachments. Attaching is offered on both
                          the public submit form and the dashboard's create-post
                          form; on plans without it the control is hidden and an
                          upgrade prompt is shown. Images are limited to 10MB and
                          video to 50MB each (duration is bounded by file size).
                          Attachments display on the post detail — photos in a
                          thumbnail grid that opens full size, video inline with
                          playback controls — and posts with media are flagged on
                          the board. Uploads are validated for type and size and
                          gated by plan on the server, independently of the UI.
    - The public portal exposes the feedback board only. Roadmap and changelog
      are NOT public (no nav entry; their portal routes redirect to the board) —
      they are managed in the admin dashboard.
    - Public responses expose only safe fields; author and guest emails are
      never returned on public routes.

3.3 Feedback Management (Dashboard)
    Feedback originates from end users on the public portal. The dashboard is
    for triage and management; the team does not author new posts here.
    FR-F1  List ......... Search and filter by status, type, and tag.
    FR-F2  Maintain ..... The team does NOT edit the text of a user's feedback —
                          the content belongs to its author. Triage is done via
                          status, tags, pin, and duplicate marking (FR-F3..F6).
                          Deleting a feedback post is restricted to the workspace
                          owner and requires a paid plan (Pro or higher); it is
                          enforced server-side and the control is hidden/locked
                          otherwise (see FR-B5).
    FR-F3  Status ....... Set status: open, planned, in progress, or completed
                          (the "closed" status is not offered in the picker).
    FR-F4  Pin .......... Pin important posts to the top.
    FR-F5  Tags ......... Assign or remove tags.
    FR-F6  Duplicates ... Mark a post as a duplicate, with suggested matches.
    FR-F7  Engage ....... Quick upvote from the list; full post view with
                          voting, tags, duplicates, and a read-only view of the
                          post's discussion. Comments are authored, replied to,
                          edited, and deleted on the public board (see FR-P5/P7),
                          not from the dashboard.
    FR-F8  Promote ...... From the Open tab, select one, several, or all posts
                          and add them to a chosen roadmap column in a single
                          action (with an optional target date). Adding a post to
                          a status column also sets its status; posts already on
                          the roadmap are skipped.
    - Guest submissions appear as "Anonymous" (or the supplied name); the team
      can see a guest's email, when provided, for follow-up.

3.4 Roadmap
    FR-R1  Columns ...... Rename and reorder columns. (Columns cannot be created
                          or deleted from the board — each workspace is seeded with
                          the default status-linked columns.)
    FR-R2  Items ........ Add a post to a column with an optional target date;
                          move items between columns by drag-and-drop; remove items.
    FR-R3  Status sync .. Columns whose key matches a post status (planned,
                          in_progress, completed, ...) are linked to that status:
                          moving an item into such a column sets the post's status,
                          and changing a post's status moves its roadmap item into
                          the matching column. Removing (dismissing) a post from the
                          roadmap returns it to the Open state, so it reappears in
                          the Open tab. The roadmap and the feedback board's status
                          tabs therefore stay consistent.

3.5 Changelog
    FR-C1  Authoring .... Create and edit entries with Markdown content.
    FR-C2  Publishing ... Save as draft or publish; only published entries are
                          visible on the portal.

3.6 Notifications
    FR-N1  In-app notifications with an unread count and deep links to the
           referenced post or changelog; mark read, mark all read, and delete.
    FR-N2  Email — the workspace owner is emailed each time new feedback is
           posted on their public board. The message shows the type, author,
           title and an excerpt, and links straight to the post in the dashboard.
           Owners are not emailed about their own submissions. Delivery never
           blocks or fails the visitor's submission.

3.7 Administration (Settings)
    Tabs: Profile, Team (roles), Tags, Integrations, API Keys, Audit Logs,
    Branding (logo, primary color, subdomain/custom domain), and Billing.
    Admin-only tabs are gated by role.

3.8 Analytics
    FR-A1  Overview of totals, status/type breakdowns, and a recent-activity trend.

3.9 Platform Services
    Scoped API keys, audit logging, third-party integrations, and file uploads.

3.10 Billing & Subscriptions
    Each workspace is on a subscription plan; payment is handled by Stripe
    (hosted Checkout + Customer Portal — no card data is stored by the platform).
    FR-B1  Plans ........ Three monthly tiers: Free ($0), Pro ($19), and Business
                          ($49). Tiers and features are presented on a public
                          pricing page and on the dashboard Billing tab.
    FR-B2  Upgrade ...... An owner/admin upgrades from the Billing tab via Stripe
                          Checkout; on success the workspace's plan is updated.
    FR-B3  Manage ....... Subscribers open the Stripe Customer Portal to change
                          tier, update card, or cancel; cancellation returns the
                          workspace to Free.
    FR-B4  Sync ......... A Stripe webhook keeps the workspace plan and status in
                          sync with the subscription lifecycle.
    FR-B5  Enforcement .. Paid capabilities are gated by plan: a Free workspace
                          cannot set a custom domain, create integrations, delete
                          feedback posts, or attach photos/video to feedback, and
                          is prompted to upgrade. Deleting feedback additionally
                          requires the owner role.
                          Team size is capped by the plan's seat limit, counted
                          as members BESIDES the owner: Free allows the owner plus
                          2 team members, Pro the owner plus 5, and Business an
                          unlimited team. The cap is enforced when an invitation is
                          sent (current members + outstanding invites). CONCURRENT
                          DEVICES are also a plan capability: Free and Pro allow a
                          single active session at a time, while Business allows
                          unlimited devices, browsers and tabs (see FR-S1, FR-S2).
                          The plan is set only by Stripe (or an admin comp) and
                          cannot be changed directly by the client.
    FR-B6  Promo codes . Admins generate promo codes (see 3.11): percentage-off
                          codes (for any or a specific plan, backed by Stripe) and
                          free-plan comps (Pro/Business). An owner redeems a code
                          in the Billing tab — free-plan codes grant the plan
                          instantly with no charge, percentage-off codes apply at
                          checkout. Codes support a duration, a redemption limit,
                          and an expiry, and may be revoked.
    FR-B7  Offers ...... Admins set a promotional price on a paid plan (see 3.11),
                          optionally time-bounded. While an offer is active, every
                          workspace's Billing tab shows that plan's list price with
                          a diagonal strikethrough beside the lower offer price and
                          a "save %" badge, and the discount is applied
                          automatically at checkout. One active offer per plan.

3.11 Platform Administration (Admin Panel)
    A platform admin manages the entire platform from a dedicated panel.
    FR-AD1 Access ...... Admins are stored separately from tenant users and sign
                          in at a dedicated admin sign-in; the admin session has
                          no workspace and is routed to the Admin Panel.
    FR-AD2 Workspaces .. View every workspace (owner, plan, status, member/post
                          counts); rename, activate/deactivate, delete, and
                          grant/revoke a plan (a comp — a paid plan with no
                          charge). Open any workspace to moderate its feedback
                          posts — change status, pin/unpin, or delete — and
                          delete any of a post's comments (comment text is not
                          editable — it belongs to its author).
    FR-AD3 Users ....... View and search all users across workspaces; change role
                          (owner/user), reset password, activate/deactivate, or
                          delete.
    FR-AD4 Admins ...... List, create, activate/deactivate, and remove other
                          admins (an admin cannot modify their own account).
    FR-AD5 Overview .... Platform-wide counts (workspaces, users, posts, paid
                          subscriptions, promo redemptions) and a plan breakdown.
    FR-AD6 Promo/Offers  Generate and revoke promo codes, and create/end
                          promotional plan offers (FR-B6, FR-B7).

--------------------------------------------------------------------------------
4. EXTERNAL INTERFACES
--------------------------------------------------------------------------------

4.1 REST API
    - JSON over HTTP; responses share a { status, message, data } envelope.
    - Authenticated routes require Authorization: Bearer <token>.
    - Public portal routes (/public/*) are unauthenticated and tenant-scoped by
      subdomain or custom domain.

4.2 User Interface
    - Portal: board, post detail, roadmap, changelog.
    - Dashboard: overview, feedback, roadmap, changelog, notifications, settings.
    - Admin Panel: overview, workspaces, users, admins, promo codes, offers.
    - A single, consistent visual system (see 5.5).

--------------------------------------------------------------------------------
5. NON-FUNCTIONAL REQUIREMENTS
--------------------------------------------------------------------------------

5.1 Performance .... Interactions feel instant; route transitions stream a
                     skeleton; API responses are compressed.
5.2 Security ....... HTTPS; JWT authentication; server-side validation; strict
                     tenant isolation; no private fields (e.g. emails) on public
                     routes.
5.3 Scalability .... Stateless API over a connection-pooled database; designed
                     for many tenants and wildcard-subdomain routing.
5.4 Availability ... Target 99.9% uptime.
5.5 Usability ...... Light-only theme; one type family; a rose product palette
                     with a per-tenant brand color on each portal; consistent
                     forms, dialogs, and dropdowns.

--------------------------------------------------------------------------------
6. DATA MODEL (principal tables)
--------------------------------------------------------------------------------

    tenants, users (role owner/user), invitations, posts (with guest submitter
    fields), post_attachments (photo/video on feedback, Pro+), votes
    (supporting guest voting), comments, tags, post_tags,
    roadmap_columns, roadmap_items, changelog_entries, notifications, api_keys,
    audit_logs, integrations, oauth_accounts. All feature tables are scoped by
    tenant. user_sessions records the active device sessions behind an account
    and backs single-session login (FR-S1). Platform-level (not tenant-scoped):
    admins (separate operator accounts), promo_codes, promo_redemptions, and
    offers.

--------------------------------------------------------------------------------
7. DEPLOYMENT
--------------------------------------------------------------------------------

    Wildcard DNS (*.<root-domain>) plus optional per-tenant custom domains;
    cloud-hosted web application and API; managed MySQL database.

--------------------------------------------------------------------------------
8. FUTURE ENHANCEMENTS
--------------------------------------------------------------------------------

    - OAuth sign-in (Google / GitHub / Microsoft).
    - Email notifications to guests who shared an address.
    - AI-assisted prioritization and duplicate detection.
    - Slack / Discord integrations.
